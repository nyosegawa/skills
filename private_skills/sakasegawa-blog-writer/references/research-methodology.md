# 調査・掘り下げ方針

## 調査の思考プロセス

トピックが与えられたとき、以下の順序で考える。

### 1. 「なぜこれが今大事なのか」を特定する

技術トピックには必ず「今これが注目されている理由」がある。それを見つける。

例:
- Context Engineering → AI Agentの実運用でコスト・精度のボトルネックになっている
- Prompt Caching → コストが1/10になるため、Agentの商用利用に不可欠

### 2. 定量データを集める

読者が一番知りたいのは「具体的にどれくらい違うのか」。数値は最優先で調査する。

優先度:
1. 公式の価格表(API pricing pages)
2. ベンチマーク結果(公式 or 再現可能なもの)
3. 実装上の数値(Context Window制限、バッファサイズ等)
4. 自分で計算した比較表(条件を明示した上で)

### 3. 実装を見に行く

概念の説明だけでなく、実際のOSSがどう実装しているかを調べる。

調査先の優先順位:
1. 記事のトピックに直接関係するOSS (GitHubリポジトリ)
2. 主要なフレームワークの該当コード
3. 公式ドキュメントの実装ガイド

例: Coding AgentのContext管理 → Codex, gemini-cli, Cline, OpenCodeの実装を比較

### 4. 論文を探す

学術的なバックグラウンドがあるトピックでは論文を引用する。

探し方:
- arXivで関連キーワード検索
- 既知の論文のReferencesから辿る
- 最新のサーベイ論文を見つける

引用の仕方:
- [著者名 et al., 年](arXivリンク) の形式
- 論文の内容は噛み砕くが、正確性は犠牲にしない
- 全部は紹介しない。記事の流れに合うものだけ選ぶ

### 5. 「掘り下げるか、Appendixに回すか」の判断

判断基準:
- 本編の流れに必要 → 本編で掘り下げる
- 興味深いが脱線になる → Appendixで扱う
- 前提知識として必要だが詳細は長い → Appendixで扱い、本編から参照

例:
- KV Cacheの仕組み → 本編で「なぜキャッシュできるか」を簡潔に → Appendixで PagedAttention, APC, RadixAttention の詳細

## 掘り下げのパターン

### パターンA: コスト分析
1. 公式価格を表にする
2. 具体的なシナリオ(例: 30ループのCoding Agent)で計算
3. キャッシュあり/なしの比較表を作る
4. 削減率を算出する

### パターンB: 実装比較
1. 複数のOSSの該当実装を調査
2. それぞれのアプローチを簡潔に記述
3. 表で比較する(例: Context Window制限の表)
4. 共通点と差異を指摘する

### パターンC: 技術原理の解説
1. 「何が問題か」から始める
2. 「なぜこの解決策が成立するか」を説明
3. 具体的な仕組みを段階的に説明
4. 限界や課題を指摘する

### パターンD: 論文紹介
1. 本編の文脈で「こういう課題がある」と提示
2. 関連する論文を2-4本程度紹介
3. それぞれの論文のキーアイデアを1-2文で説明
4. 全体的なトレンドをまとめる

## 調査で避けること

- 網羅的な比較(すべてのツールを並べる等)。記事の主張に関係するものだけ
- 公式ドキュメントのコピペ。読者は公式を読めばいいのであって、記事に求めているのは咀嚼された理解
- 古い情報の引用。価格やAPIは特に変わりやすいので最新を確認する
- 未検証の情報。「〜らしい」は避ける。確認できないものは書かない
