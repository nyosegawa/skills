# 文体・フォーマットガイド

## 基本トーン

カジュアルな語り口で、読者に直接話しかけるような文体。ただしふざけすぎない。技術的な正確さは絶対に維持する。

OK: 「めっちゃ安いからです (結論)」「嬉しみがあります」「クソデカJSONだったりもする」
NG: 全文ふざける。技術的に不正確なジョークを入れる

## 記事冒頭

記事の冒頭は必ず以下のパターンで始める。

```
こんにちは！逆瀬川ちゃん ([@gyakuse](https://x.com/gyakuse)) です！

今日は{トピック}について{何をするか}をまとめていきたいと思います。
```

「まとめていきたいと思います」の部分は、記事の内容に応じて「見ていきたいと思います」「考えていきたいと思います」等に変える。

## 句読点ルール

### 読点(、)
音読したときのリズムに合わせて打つ。無闇に打たない。

OK: 明日は晴れたらいいですね。そうすればピクニックにも行けるし、遊園地だって遊びに行けるし
NG: 明日は、晴れたらいいですね。そうすれば、ピクニックにも行けるし、遊園地だって、遊びに行けるし

主語の直後に読点を打たないことが多い。読んだときに息継ぎが必要な箇所に打つ。

### 句点(。)
リスト項目の末尾には句点をつけない。

OK:
- Context EngineeringはCacheが要です。Cacheコントロールなしには成立しません

NG:
- Context EngineeringはCacheが要です。Cacheコントロールなしには成立しません。

ただし通常の文章(リスト外)では普通に句点を使う。

## フォーマットルール

### 太字(bold)
基本的に使わない。本文中で太字強調は使わない。

OK: Agentには環境別に考えると、大まかにわけて以下の4つのタイプがあると思います
NG: **Agent**には環境別に考えると、大まかにわけて以下の4つのタイプがあると思います

### 英単語周りのスペース

英単語の前後にスペースを入れるかは気分次第。毎回統一しない。

OK (混在してよい):
- Context Engineeringは非常に面白いです
- とくにCache の考え方は魅力的です
- これはvLLM等でセルフホスティングした場合も同様です

NG (全部揃えようとする):
- Context Engineering は非常に面白いです。とくに Cache の考え方は魅力的です

自然に見えることが大事。機械的にルールを適用しない。

### 見出し
`##` で章、`###` で節。`#` は使わない(タイトルはフロントマターの `title` で指定する)。
見出しに太字は使わない(Markdownの見出し記法だけで十分)。

### 表
比較データは積極的に表にする。表は説得力がある。
価格比較、性能比較、機能比較などは表で見せる。

### コードブロック
実在のOSSからの引用が望ましい。引用元のURLをコードブロックの前後に記載する。

### 画像
必要に応じて `![](URL)` で画像を挿入する指示を残す。自分で画像は生成しない。「ここに図を入れる」等の注記を入れてもよい。

## 章構成ルール

### 章間の接続(最重要)

前の章の最後の話題が次の章の冒頭で拾われること。これが読みやすさの核心。

例:
- 前章の末尾: 「...これがContext CachingやPrompt CachingやPrefix Cachingの基本原理です。」
- 次章の冒頭: 「さて、キャッシュ化はされたのですが、うまく使われなきゃいけません。」

接続に使える表現:
- 「さて、このように〜を見たときに」
- 「ここで〜を振り返っていきましょう」
- 「このあたりは実は〜にもやっているのですが」
- 「さて、この〜を見ることで〜が見えてきます」

### 導入パターン

概念を導入するとき、定義をサラッと流してすぐ具体例に移る。

OK:
```
定義としては
- 推論において最適な情報群をキュレートしたり維持する一連の戦略 (Anthropic)
などが多くわりとふわっとしているのであんまり定義を振り返ることはありません。
{具体的な構造}を考えるためには、{実例}を考えるのが一番です。
```

NG: 定義を3段落かけて丁寧に説明する

### 結論の提示

結論を先に出してから理由を説明するパターンを使うことがある。

例: 「めっちゃ安いからです (結論)」

### まとめ

まとめは短くする。2-3個のbulletで十分。

```
## まとめ

- {最も重要なポイント}
- {次に重要なポイント}
```

長々と振り返らない。

### Appendix

本編で深入りしすぎると流れが悪くなる技術的詳細はAppendixに逃がす。本編から「Appendixを参照してください」とリンクする。

Appendixは本編より技術的に深くてよい。論文の詳細、アルゴリズムの仕組み、実装の詳細などをここに書く。

### References

記事の末尾にリンク集を置く。カテゴリ分けしてもよい。URLは必ずMarkdownリンク形式 `[タイトル](URL)` で記述する。生URLをそのまま書かない。

```
## References

- [ドキュメント名](https://example.com/doc1)
- Paper
    - [論文タイトル](https://arxiv.org/abs/xxxx.xxxxx)
- Pricing
    - [サービス名 Pricing](https://example.com/pricing)
```

### URL・リンクの書き方

記事中のすべてのURLはMarkdownリンク形式にする。これはReferencesに限らず、本文中のリンクもすべて同じ。

OK: [Anthropic API Pricing](https://docs.anthropic.com/en/docs/about-claude/pricing)
NG: https://docs.anthropic.com/en/docs/about-claude/pricing

## 文末ルール

### 敬体(です・ます調)

本文は一貫して敬体で書く。常体(だ・である調)を混ぜない。

### 体言止め禁止

本文中の体言止め(名詞や名詞句で文を終えること)は禁止。必ず「です」「ます」等の敬体で終える。

OK: ワンタップで承認できます。
NG: ワンタップで承認。

OK: コピペで使えるようにしています。
NG: コピペで使えるように。

ただし見出し、表のセル、リスト項目内では体言止めOK。

## やってはいけないこと

- 太字を多用すること
- 読点を機械的に打つこと
- 定義の説明に長い時間をかけること
- ふざけた表現を完全に排除すること
- 章を独立して書いて後からつなげようとすること(最初から流れを意識して書く)
- すべての英単語の前後にスペースを入れること
- リスト末尾に句点をつけること
- URLを生のまま(Markdownリンクにせずに)記述すること
- 本文で体言止めを使うこと(見出し、表、リスト内はOK)
